# Backpacker Product Overview

Версия продукта: `1.1.2.17`
Статус: первая стабильная линия, активная версия с PDF-экспортом, read-only шарингом, received-списком, предложениями доли и новых идей, минимальным shared-профилем и iPhone PWA readiness
Дата: 2026-07-05

## One-liner

Backpacker — мобильный личный планировщик поездки для самостоятельных путешественников, который собирает идеи, брони, места, события, ссылки и расходы в одну управляемую карточную поездку.

## Product Promise

Backpacker помогает планировать поездку без хаоса из заметок, таблиц, чатов и вкладок. Пользователь быстро складывает все элементы поездки в карточки, раскладывает их по дням, видит бюджет, сохраняет запасные идеи и может поделиться планом или сметой.

## Target User

Основной пользователь — самостоятельный путешественник, который планирует поездку сам: ищет билеты, отели, кафе, экскурсии, необычные места, бани, прогулки, запасные варианты и хочет держать все в одном месте.

Типичный сценарий:

- поездка планируется постепенно;
- источников много: сайты, мессенджеры, карты, заметки, рекомендации;
- часть идей еще не решена;
- бюджет важен, но не должен превращать продукт в бухгалтерию;
- телефон — главный рабочий экран во время поездки.

## Core Job To Be Done

Когда я планирую поездку из множества мест, броней, идей и расходов, я хочу быстро собрать все в один живой план по дням, чтобы не терять важное, видеть бюджет и спокойно менять маршрут без переписывания таблиц.

## Current Product Shape

Backpacker V1.1.2.17 состоит из основных зон:

- Главная: тренажер, создание поездки, список поездок, быстрые инфо-действия.
- `Со мной поделились`: отдельный read-only список поездок, которые получатель сохранил из публичной ссылки.
- Минимальный shared-профиль: имя или ник текущего anonymous user для совместных сценариев, без регистрации и без входа по имени.
- Shared-предложения: получатель read-only ссылки может предложить долю в расходе или новую идею для поездки, а автор модерирует предложения.
- План по дням: дорожки дней, карточки событий, парковка идей.
- Корзина/идеи: группировка карточек по статусам и запасным вариантам.
- Бюджет: расходы, оплачено, возможный итог, остаток.
- Валюты и экспорт: калькулятор валют, PDF/XLS выгрузки, текстовый share-сценарий, PDF-сценарий поездки и read-only ссылка на опубликованную поездку.
- PWA-режим: установка на домашний экран iPhone и standalone-запуск без интерфейса браузера.

## Key Capabilities

### Trip Basket

Пользователь может создать поездку, задать направление, даты, валюту и бюджет. Личные нерасшаренные поездки хранятся локально в браузере. В Supabase отправляется только snapshot поездки, для которой автор включил доступ по ссылке.

### Cards

Карточка хранит:

- название;
- тип;
- статус;
- приоритет;
- дату;
- время;
- длительность;
- цену;
- оплаченную сумму;
- участника/владельца расхода;
- ссылку;
- локацию;
- заметку.

Карточка остается валидной даже без даты, времени или длительности.

### Day Tracks

Дни поездки представлены как дорожки. Карточки можно раскладывать по дням, менять порядок и переносить между днями. Поведение drag/drop, автоскролла и мобильных свайпов считается каноническим для V1.1.0.0.

### Budget

Бюджет показывает:

- общий лимит поездки;
- уже оплачено;
- обязательные траты;
- опциональные траты;
- возможный итог;
- остаток.

Цель бюджета — быстро понять картину поездки, а не заменить полноценный финансовый учет.

### Export

Пользователь может выгрузить:

- смету;
- план по дням;
- PDF поездки;
- XLS/Excel.

В XLS время фиксируется как текст, чтобы Excel не конвертировал его в дату. В смете есть итоговая строка по денежным колонкам.

PDF поездки собирается локально из данных приложения. В `Поделиться поездкой` пользователь может отправить поездку текстом, открыть настройки PDF, выбрать включение сметы, заметок и элементов без даты, а затем отдельно нажать `Скачать` или `Поделиться`. `Скачать` сохраняет файл локально; `Поделиться` вызывает системное меню файлового share, а если устройство его не поддерживает, использует скачивание как fallback.

Read-only ссылка создается через Supabase Edge Function. Автор технически определяется Supabase Anonymous Auth без отдельного экрана регистрации. Гость открывает ссылку без пользовательской сессии и видит только просмотр. Опция `Показывать смету` управляет серверной проекцией: при выключении финансовые поля удаляются на стороне Edge Function, а не только скрываются в интерфейсе.

Минимальный профиль в `1.1.2.12` хранит только `display_name`. Это имя или ник используется для отображения в shared-сценариях, но не является логином, не уникально и не участвует в проверке прав. Права и ownership по-прежнему определяются только через Supabase `auth.uid()`. Обычная локальная работа с поездками не требует имени; prompt появляется перед первым shared-действием, где имя будет видно другой стороне. Email, телефон, OAuth, аватар, контакты, восстановление доступа и cloud sync личных поездок не добавлены.

В `1.1.2.13` получатель read-only ссылки может предложить автору новую идею через существующую кнопку `+`. Pending-идея хранится в `trip_share_item_proposals`, не является `TripItem`, не попадает в опубликованный snapshot и не влияет на бюджет, смету, PDF или Excel. После принятия автором идея атомарно превращается в обычную карточку в разделе `Без даты`; если у идеи указана цена, расход полностью назначается автору по текущей модели allocations. На принятой карточке показывается вторичный маркер `Предложено: Имя`, где имя является display snapshot на момент принятия и не используется для прав или идентификации.

### Путешествуйте с друзьями

Автор может открыть поездку, нажать `Поделиться поездкой`, выбрать `Открыть доступ по ссылке` и скопировать публичную ссылку. В Supabase отправляется только эта опубликованная поездка; остальные личные поездки остаются в `localStorage` и не синхронизируются.

Гость открывает ссылку без отдельной регистрации и видит read-only версию поездки. Если хочет вернуться к ней позже, он нажимает `Добавить в «Со мной поделились»`; на главной появляется отдельная карточка приглашенной поездки. Карточка показывает статус, автора, название, направление, даты, количество дней и бюджетную плашку. Сейчас бюджетная плашка показывает бюджет поездки, если автор разрешил смету; доля конкретного гостя остается отдельной продуктовой задачей.

Гость не редактирует поездку напрямую. Он может:

- предложить взять на себя часть расхода, если смета открыта;
- предложить новую идею, место или событие;
- отозвать pending-предложение до решения автора.

Автор видит предложения внутри поездки в блоке `Предложения`, принимает или отклоняет их. До принятия предложения не попадают в план, бюджет, PDF, Excel или опубликованный snapshot. После принятия предложение расхода меняет распределение расхода, а предложение идеи становится обычной карточкой поездки.

Ограничения текущего MVP:

- Backpacker сейчас в бета-версии: надежнее планировать поездку и пользоваться приложением с одного основного устройства;
- нет постоянных аккаунтов, email, телефона, OAuth и восстановления доступа;
- shared-действия привязаны к текущей Supabase Anonymous Auth session;
- ноут, мобильный браузер и установленное PWA пока могут считаться разными пользователями;
- после очистки данных, переустановки PWA или потери anonymous session управление старой ссылкой и список `Со мной поделились` могут не восстановиться;
- имя или ник используется только для отображения другой стороне и не участвует в правах;
- при выключенной смете финансовые поля удаляются серверной read-only проекцией.

### Установка на iPhone

Для iPhone Backpacker можно установить на домашний экран как PWA: открыть приложение в Safari, нажать системную кнопку `Поделиться`, выбрать `На экран Домой`, включить `Открывать как веб-приложение` и нажать `Добавить`.

Рекомендуется установить Backpacker до создания первой поездки, потому что Safari и приложение с домашнего экрана используют разные локальные хранилища.

### Ближайший shared backlog

После текущего beta-релиза ближайшие улучшения shared-сценария:

- на карточке приглашенной поездки показывать долю гостя, когда у него есть accepted participant link/allocation; до этого явно подписывать общий бюджет как `Бюджет поездки`;
- дать гостю редактировать идею до отправки; для уже отправленной pending-идеи MVP-путь остается `отозвать и создать заново`, пока не появится безопасный edit-flow;
- на карточке авторской групповой поездки показывать яркий входящий маркер предложений, например торчащий сверху ярлычок `+1 идея от участника` или `Есть предложения · N`; для этого нужен легкий backend-count, чтобы главная не загружала все предложения по всем поездкам.

Канонический вид PDF в `1.1.2.6`:

- верхняя шапка повторяет композицию мобильной шапки поездки без кнопок действий: логотип, название, направление/даты/длительность, плашка бюджета и три бюджетные карточки;
- бюджетные строки в PDF соответствуют шапке поездки: `Бюджет`, `Оплачено`, `Уже распределено`, `Осталось распределить`;
- строка участников показывается только в групповой поездке, если участников больше одного;
- карточки в PDF повторяют пропорции карточек приложения и укладываются в сетку 4 карточки в ряд;
- карточки сохраняют типовые цвета, шапку, price-pill, длительность, время, дату, заметку и бейджи статуса/участника;
- если событий в дне больше четырех, они продолжаются следующей строкой ниже.

### Trainer

Тренажер — демо-поездка, в которой можно потрогать карточки, дни, бюджет и экспорт без риска испортить реальную поездку.

## UX Principles

- Mobile-first.
- Карточки важнее таблиц.
- Пустые данные не ломают карточку.
- Drag/drop не должен конфликтовать со скроллом.
- Нижние действия должны быть под рукой.
- Интерфейс должен помогать планировать, а не объяснять себя длинными инструкциями.
- Экспорт должен быть практичным: открыл, отправил, вставил в Excel.

## Stable Baseline

Первая стабильная версия зафиксирована в `BACKPACKER_STABLE_V1_1_0_0.md`.

Особо хрупкие зоны:

- drag/drop карточек;
- автоскролл при перетаскивании;
- свайпы дорожек дней;
- mobile touch balance;
- sticky actions в форме карточки;
- XLS-экспорт времени.

Любая работа в этих зонах требует ручной проверки на desktop и mobile.

## Versioning

Backpacker использует формат:

`super-release.major-epic.minor-feature.patch`

Текущая версия:

`1.1.2.17`

Расшифровка:

- `1` — первая стабильная продуктовая линия.
- `1` — первый большой эпик: личный планировщик поездки.
- `2` — мини-фича PDF/share поездки поверх стабильной базы и копирования карточек.
- `17` — семнадцатый патч внутри PDF/share-ветки: уточненные инструкции по shared-сценариям для пользователей.

## Non-goals For Current Version

В текущей стабильной линии намеренно нет:

- постоянных аккаунтов;
- совместного редактирования;
- полного CRUD всех личных поездок в backend;
- карт;
- бронирования;
- платежей;
- полноценного multi-device sync.

## Patch 1.1.0.1

Патч `1.1.0.1` сохраняет каноническую стабильную базу `1.1.0.0` и добавляет подготовленный, но публично заблокированный сценарий поддержки Backpacker. В коде есть модалка `Копилка Backpacker`, выбор суммы `100 ₽`, `300 ₽`, `500 ₽` или другой суммы, переход на внешнюю платежную страницу и аналитические события открытия сценария. Кнопка поддержки на главной остается в disabled-состоянии до отдельной команды на активацию, поэтому платежи по-прежнему не являются публичной возможностью текущей версии.

## Feature 1.1.1.0

Версия `1.1.1.0` добавляет копирование существующей карточки из редактора: копию можно положить в другой день текущей поездки, в `Без даты`, в выбранный день другой поездки или в `Без даты` другой поездки. Исходная карточка не меняется, копия получает новый идентификатор, дата задается выбранным назначением, факт оплаты сбрасывается, а при разной валюте целевой поездки стоимость не копируется. На самих карточках не появляются новые кнопки, DOM карточек и drag/drop-механика остаются канонически неприкосновенными.

## Feature 1.1.2.0

Версия `1.1.2.0` расширяет действие `Поделиться поездкой`: пользователь может отправить текущую текстовую сводку или создать локальный PDF-файл поездки. PDF собирается из данных поездки, содержит хронологический план по дням, опциональную смету, опциональные заметки и опциональный блок `Без даты`; после генерации файл отправляется через системное меню, если устройство поддерживает share файлов, иначе скачивается локально.

## Patch 1.1.2.6

Версия `1.1.2.6` фиксирует текущий канонический PDF-вид поездки. В PDF больше нет отдельной табличной адаптации карточек: карточки масштабируются из пропорций интерфейса приложения, занимают 4 колонки на странице и продолжаются дополнительными строками внутри дня. Верх PDF повторяет мобильную шапку поездки без кнопок. Действия с PDF разделены на две кнопки: `Скачать` и `Поделиться`.

## Patch 1.1.2.7

Версия `1.1.2.7` добавляет первый шаг активного шаринга: автор может открыть доступ к текущей поездке по ссылке, скопировать ее, автоматически обновлять опубликованный snapshot после сохраненных изменений и отозвать доступ. Raw share-token хранится только локально для повторного копирования, в Supabase хранится только hash. Управление ссылкой привязано к текущей anonymous-сессии браузера; после потери сессии управление существующей ссылкой не восстанавливается до появления постоянных аккаунтов.

Шаринг остается read-only для гостя. При выключенной смете Edge Function формирует серверную проекцию без финансовых полей. Личные нерасшаренные поездки не синхронизируются и остаются в `localStorage`.

## Patch 1.1.2.8

Версия `1.1.2.8` добавляет iPhone PWA readiness: корректные manifest/iOS meta tags, apple touch icon 180x180, standalone display и компактную подсказку установки на экран Домой. Подсказка показывается только в браузерном режиме на iPhone/iPad, не показывается внутри установленного приложения и скрывается после закрытия в рамках текущей версии.

В подсказке зафиксировано ограничение iOS: Home Screen web app имеет отдельный `localStorage` от Safari. Пользователю рекомендуется установить Backpacker до создания первой поездки; автоматический перенос Safari-данных не выполняется.

## Patch 1.1.2.9

Версия `1.1.2.9` добавляет раздел `Со мной поделились`. Получатель публичной ссылки может вручную сохранить read-only поездку на главной. Сохранение создаёт серверную связь `recipient_user_id + trip_share_id`, не копирует snapshot в личные поездки и не использует raw token для повторного открытия.

Сохранённые карточки отделены от личной ленты. По нажатию Backpacker читает актуальную read-only версию через `read_received`. Если автор отозвал доступ, карточка остаётся в списке со статусом `Доступ закрыт`, данные поездки не открываются, доступно удаление из списка.

MVP-ограничение: список `Со мной поделились` привязан к текущей Supabase Anonymous Auth session. После очистки данных приложения, переустановки PWA или потери anonymous session список может не восстановиться до появления обычной авторизации.

## Patch 1.1.2.10

Версия `1.1.2.10` добавляет публичный frontend Supabase config в PWA-сборку, чтобы установленное приложение могло создавать и копировать share-ссылки без локального `supabase-config.js`. Во frontend попадают только `project URL` и publishable anon key; service-role key остаётся только в Edge Function.

## Patch 1.1.2.11

Версия `1.1.2.11` добавляет первый сценарий участия получателя в read-only поездке: он может предложить автору взять часть расхода из текущей доли автора. До принятия предложения поездка не меняется.

Автор принимает предложение атомарной PostgreSQL RPC-операцией: сервер повторно проверяет pending-статус, ownership, долю автора, currency, financial version, participant link-state и invariant суммы allocations. Перед accept клиент автора отправляет свежий локальный snapshot существующим update-механизмом, чтобы не перезаписать локальную поездку старой опубликованной версией.

MVP остаётся на Supabase Anonymous Auth. Не добавляются постоянные аккаунты, email, телефон, OAuth, пользовательские профили или metadata. `proposed_participant_name` — только отображаемый псевдоним для автора конкретной поездки.

## Release Checklist

Перед релизным push:

1. Обновить `APP_VERSION`.
2. Обновить `APP_RELEASE_SUMMARY`.
3. Обновить `CACHE_NAME` в `service-worker.js`, если менялись публичные файлы приложения.
4. Прогнать `node --check app.js`.
5. Прогнать `git diff --check`.
6. Обновить `CHANGELOG.md`.
7. Для публичного demo пушить в `origin` и `demo`.
