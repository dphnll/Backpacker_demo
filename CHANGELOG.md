## 1.1.2.13 - 2026-07-03

- Добавлен Step 3: получатель read-only ссылки может предложить автору новую идею, место или событие через существующую кнопку `+`.
- До принятия идея хранится отдельно от `TripItem`, не попадает в snapshot поездки, не влияет на план, бюджет, смету, PDF или Excel.
- Получатель видит свои идеи в блоке `Мои предложения` и может отозвать pending-идею.
- Автор видит идеи в общем блоке `Предложения · N` вместе с предложениями по расходам, может принять или отклонить идею.
- Принятие выполняется атомарно через PostgreSQL RPC: идея превращается в обычный `TripItem` в разделе `Без даты`, а повторный accept не создаёт дубль.
- У принятой карточки сохраняется техническая связь с proposal и display snapshot `Предложено: Имя`; имя не участвует в правах и не обязано ретроактивно меняться после изменения профиля.
- Постоянные аккаунты, email, телефон, OAuth, комментарии, уведомления, голосования и cloud sync не добавлены.

## 1.1.2.12 - 2026-07-03

- Добавлен минимальный отображаемый профиль поверх Supabase Anonymous Auth: пользователь может указать имя или ник для shared-сценариев.
- Существующий `auth.uid()` сохраняется: share-ссылки, received-списки, proposals и participant links остаются привязаны к той же anonymous-сессии.
- Обычные локальные поездки по-прежнему доступны без имени; обязательный prompt появляется только перед shared-действиями, где имя увидит другая сторона.
- В чужой read-only поездке показывается только автор (`Автор — Миша`), без дублирования имени текущего пользователя; если профиля автора нет, используется `Автор поездки`.
- Профили хранят только `display_name`; email, телефон, OAuth, аватары, контакты и cloud sync не добавлены.
- Таблица `user_profiles` закрыта RLS: пользователь читает и меняет только собственный профиль, имена других пользователей возвращаются только через контролируемый Edge Function response.

## 1.1.2.11 - 2026-07-03

- Добавлен Step 2: получатель read-only ссылки может предложить взять часть расхода из текущей доли автора.
- Предложение может ссылаться на существующего несвязанного участника или содержать отображаемый псевдоним нового участника.
- Автор видит минимальный блок `Предложения · N` и может принять или отклонить pending-предложение.
- Принятие выполняется атомарно через PostgreSQL RPC: связывает участника, переносит allocation, обновляет snapshot и меняет статус предложения одной транзакцией.
- Сервер запрещает предложения при `includeBudget=false`, проверяет financial version и переводит устаревшие предложения в `stale`.
- MVP остаётся на Supabase Anonymous Auth: без постоянных аккаунтов, email, телефона, OAuth, профилей и пользовательских metadata.

## 1.1.2.10 - 2026-07-03

- Service-role key по-прежнему не попадает во frontend; он используется только внутри Supabase Edge Function.

## 1.1.2.9 - 2026-07-02

- Добавлен Step 1.1 `Со мной поделились`: получатель может вручную сохранить публичную read-only поездку в отдельный список на главной.
- Сохранение создаёт связь `recipient_user_id + trip_share_id` в Supabase и не копирует snapshot поездки в `localStorage`.
- Сохранённая поездка открывается с главной через `read_received` по проверенной связи, без raw share-token.
- Повторное добавление той же поездки недоступно; автор не может добавить собственную ссылку в received-список.
- Если автор отозвал доступ, карточка остаётся в блоке и показывает `Доступ закрыт`; получатель может удалить её из своего списка.
- Зафиксировано MVP-ограничение: список зависит от текущей Anonymous Auth session и может не восстановиться после очистки данных, переустановки PWA или потери anonymous session.

## 1.1.2.8 - 2026-07-02

- Добавлена iPhone PWA readiness: manifest получил `id`, standalone-настройки проверены, добавлена iOS touch icon 180x180 и iOS meta tags.

## 1.1.2.7 - 2026-07-02

- Добавлен первый шаг живого шаринга поездки через Supabase: автор может открыть доступ по ссылке, скопировать ссылку, обновлять опубликованную поездку после сохранений и отозвать доступ.
- Публичная ссылка открывает поездку в read-only режиме без авторизации гостя.
- Настройка `Показывать смету` управляет серверной проекцией: при выключении финансовые поля не отдаются из Edge Function.
- Личные нерасшаренные поездки остаются только в `localStorage`; в Supabase отправляется только опубликованный snapshot конкретной поездки.
- Управление ссылкой привязано к текущей anonymous-сессии автора. После потери anonymous-сессии управление существующей ссылкой не восстанавливается до появления постоянных аккаунтов.
- Raw share-token хранится только локально для повторного копирования; в Supabase хранится только hash.
- UI шаринга уточнен: деталка ссылки открывается под кнопкой `Открыть доступ по ссылке`, повторный клик скрывает ее, кнопка копирования приведена к хромовому стилю.
